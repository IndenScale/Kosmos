# 4a. 深度解析：Parser 的工作原理

在 Kosmos 的文档摄入流程中，“解析”（Parsing）是将原始文件转化为结构化 `Fragment` 的第一步，也是实现多模态知识理解的基础。`app/parsers/` 目录下的解析器模块通过一个精巧的工厂模式和针对性的处理策略，高效地处理各种类型的文档。

## 解析器工厂 (ParserFactory)

所有解析任务的入口点是 `ParserFactory`。它的核心职责是“诊断”文件类型，并为其分配合适的“专科医生”（即具体的解析器）。

1.  **MIME 类型检测**: 首先，工厂使用 `python-magic` 和 `mimetypes` 库来检测文件的 MIME 类型。
2.  **解析器选择**: 工厂会按照预设的优先级（从最特殊到最通用）遍历所有已注册的解析器，并调用每个解析器的 `can_parse()` 方法。第一个返回 `True` 的解析器将被选中来处理该文件。例如，`MarkdownParser` 的优先级高于 `PlainTextParser`，确保 `.md` 文件能被正确地按标题结构进行解析。

## 三类核心解析流程

根据文档的复杂性，我们可以将解析流程分为三大类：纯文本、独立图像和图文混合文档。

```mermaid
graph TD
    A[上传文件] --> B{ParserFactory};
    B -- .txt, .md, .py --> C[纯文本/代码解析流程];
    B -- .png, .jpg --> D[独立图像解析流程];
    B -- .pdf, .docx --> E[图文混合文档解析流程];

    subgraph "纯文本/代码 (Plain Text / Code)"
        C --> C1[读取全部文本内容];
        C1 --> C2[智能文本分割];
        C2 --> C3[创建 Text Fragments];
    end

    subgraph "独立图像 (Image)"
        D --> D1[调用 VLM 分析图像];
        D1 --> D2{生成文本描述};
        D2 --> D3[创建 Figure Fragment (图像本身)];
        D2 --> D4[创建 Text Fragment (文本描述)];
    end

    subgraph "图文混合 (PDF-like)"
        E --> E1[分层内容提取];
        E1 --> E2[提取原始文本流];
        E1 --> E3[提取内嵌图片];
        
        E1 --> E4[为每页创建 Screenshot Fragment]

        E3 --> E5[调用 VLM 分析图片];
        E5 --> E6{生成图片描述};
        E6 --> E7[创建 Figure Fragments];

        E2 & E6 --> E8[将图片描述插回文本流];
        E8 --> E9[智能文本分割];
        E9 --> E10[创建 Text Fragments];
    end
```

### 1. 纯文本/代码解析流程

-   **适用对象**: `.txt`, `.md`, `.py`, `.js` 等不包含二进制图像内容的文件。
-   **核心逻辑**:
    1.  **完整读取**: 解析器首先将文件的全部内容读入内存。
    2.  **智能分割**: 这是关键步骤。内容会被送入一个**智能文本分割器 (`TextSplitter`)**。
        -   对于 **Markdown (`.md`)**，分割器会优先尝试按**标题层级**进行分割，确保每个 `Fragment` 都是一个逻辑完整的章节。
        -   对于**代码文件**，分割器会尝试按**类或函数**定义进行分割。
        -   对于**普通文本 (`.txt`)** 或过长的 Markdown/代码块，分割器会采用基于句子语义边界的算法（如递归字符分割），在保证每个 `Fragment` 不超过设定长度（如 1500 字符）的同时，尽可能避免将一个完整的句子切开。
    3.  **产出**: 生成一系列 `text` 类型的 `Fragment`。

### 2. 独立图像解析流程

-   **适用对象**: `.png`, `.jpg`, `.webp` 等独立的图像文件。
-   **核心逻辑**:
    1.  **VLM 分析**: 图像文件被直接发送给**视觉语言模型 (VLM)**。
    2.  **双重产出**:
        -   **Figure Fragment**: 创建一个 `figure` 类型的 `Fragment`，其 `raw_content` 字段存储了处理后（如统一格式化、缩放）的图像路径。这代表了图像本身。
        -   **Text Fragment**: VLM 返回的**文本描述**被用来创建一个独立的 `text` 类型的 `Fragment`。这个 `Fragment` 的内容是 VLM 对图像的理解（例如，“一张柱状图，显示了公司四个季度的销售额…”）。
-   **价值**: 通过这个流程，一张原本不可搜索的图片被转化为了一个可被索引和语义搜索的文本片段，实现了真正的多模态知识转化。

### 3. 图文混合文档解析流程 (以 PDF 为例)

-   **适用对象**: `.pdf`, `.docx` 等复杂的、同时包含大量文本和嵌入式图片的文件。这是最复杂但也是最强大的解析流程。
-   **核心逻辑**:
    1.  **分层内容提取**: 解析器（如 `PdfParser`）会遍历文档的每一页，同时提取两种类型的内容：
        -   **原始文本流**: 提取所有可识别的文本，并按页面顺序拼接起来。
        -   **内嵌图片**: 提取所有独立的图片对象，并将其保存为临时文件。
    2.  **创建截图 (`Screenshot`)**: 作为上下文的“快照”，解析器会为文档的**每一页**都生成一张截图，并创建一个 `screenshot` 类型的 `Fragment`。
    3.  **并行处理图片**: 所有提取出的内嵌图片被并行发送给 **VLM** 进行分析，生成各自的文本描述。
    4.  **创建图表 (`Figure`)**: 每张内嵌图片都会被创建为一个 `figure` 类型的 `Fragment`，其 `raw_content` 存储了图片路径。
    5.  **文本流增强**: VLM 返回的图片描述并不会直接作为独立的 `text` 片段。相反，它们会被**插回**到第一步提取的原始文本流中，替换掉图片原来的位置占位符。
    6.  **最终分割**: 这个经过“增强”的、既包含原文又包含图片描述的完整文本流，最后被送入**智能文本分割器**，分割成一系列 `text` 类型的 `Fragment`。
-   **价值**:
    -   **上下文完整性**: 将图片描述插回原文，使得最终生成的 `text` 片段保持了图文上下文的连续性。一个 `Fragment` 可能既包含描述图片前的一段话，也包含图片本身的 VLM 描述，以及图片后的结论段落。这对于后续的语义理解和问答任务至关重要。
    -   **多模态融合**: 这种方法不是简单地将文本和图片分开处理，而是将视觉知识（通过 VLM）无缝地融合进了文本知识流中，实现了更高层次的多模态知识整合。
